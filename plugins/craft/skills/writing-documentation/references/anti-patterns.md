# Anti-patterns catalog

The 31 failure modes that show up most often in technical documentation. Three categories: structural, voice and tone, and AI-generated prose telltales.

Each entry lists what the anti-pattern looks like, why it fails, and how to fix it. The fix column is the part to remember; the rest is context.

## Structural anti-patterns

| # | Anti-pattern | What it looks like | Why it fails | How to fix |
|---|---|---|---|---|
| S1 | Tutorial-as-reference | A reference page opens with a step-by-step walkthrough before defining anything. | Mixes two reader modes: the learner doing a guided exercise vs the practitioner consulting a fact. The walkthrough reader feels lost without context; the reference reader cannot find the fact they came for. | Physically separate the content. Put the walkthrough in a Tutorial page, the facts in Reference. Link between them. |
| S2 | Reference-as-tutorial | API docs that read like a step-by-step lesson, with preamble, scaffolding, and summaries. | Reference is consulted, not read. Adding narrative buries the spec under pedagogy. | Strip all pedagogical framing from reference. State facts. Use tables, lists, definitional sentences only. |
| S3 | Walls of text | Paragraphs of 8 to 15 or more lines, no sub-headings, no lists, no white space between ideas. | Readers scan in F-shapes; impenetrable blocks kill comprehension before the first sentence is finished. Microsoft: "three to seven lines is about the right length for a paragraph." | Hard limit of five rendered lines per paragraph. Each paragraph covers one idea. Sub-headings every two to four paragraphs in long sections. |
| S4 | Mixed-level navigation | H2 "Installation" alongside H2 "Why we built this product". One is a task, one is marketing. | Readers use heading hierarchies to build a mental model of scope. Peer headings at the same level signal equal cognitive weight. Mismatched peers destroy the navigation contract. | Apply Diatáxis. Each heading belongs to exactly one mode. Never mix modes at the same heading level. |
| S5 | Burying the lede | Three paragraphs of background before stating what the page actually does. | First-screen content is most likely to be read. The answer the reader wants is usually in paragraph four. | Move the direct answer or outcome to the first sentence or first bullet. Background goes after the payoff, not before. |
| S6 | Over-tabulation | A two-column table for a list of three things that have no meaningful dimensions to compare. | Tables impose cognitive overhead: parsing row headers, column headers, cell relationships. For a list with no comparative dimension, a plain bulleted list costs less attention. | Default to prose or bullets unless comparing at least two attributes across at least three items. |
| S7 | Under-tabulation | Five paragraphs comparing three deployment options across four criteria, written as prose. | The reader must hold all prior cells in working memory to evaluate each new cell. This is exactly the job tables exist to do. | Extract the attributes (rows) and options (columns). Build the table. Replace the prose with a one-sentence intro. |
| S8 | Inconsistent heading voice | H2 "Configuration" / H3 "Set up the database" / H3 "Authentication options" / H3 "Running the server". | Noun-phrase and imperative headings signal completely different document types. Mixing them within a section destroys the implicit promise each heading makes. | Choose one style per document type. How-to: imperative. Reference: noun phrases. Explanation: gerund or "About X". |
| S9 | Clever-first code samples | Showing the idiomatic one-liner with full error handling, optional flags, and context variables as the first example. | Diatáxis tutorial principle: "ruthlessly minimise explanation." A maximally clever sample requires the reader to parse syntax they do not know yet before they can execute anything. | Show the simplest possible version first. Annotate with comments for each conceptual step. Put the production-ready version in a "next steps" section. |
| S10 | Doc-type contamination | Tutorial sections that explain theory; how-to guides that teach background; reference pages that instruct. | Each contamination removes utility from the reader who came for the pure type. | After drafting, classify every paragraph. If it does not match the document's declared mode, cut or move it. |

## Voice and tone anti-patterns

| # | Anti-pattern | What it looks like | Why it fails | How to fix |
|---|---|---|---|---|
| V1 | Hedge inflation | "Perhaps you might want to consider enabling this option, which could be fairly useful in certain cases" | Hedges signal authorial uncertainty: appropriate in research disclaimers, devastating in instructions. The reader cannot tell what is optional from what is required. | Eliminate every hedge that is not a genuine technical qualifier. "This option may cause issues on Windows" is a real conditional. "Perhaps consider" is noise. |
| V2 | Operator phrases | "In order to configure the service...", "due to the fact that...", "at this point in time..." | Orwell identified these as "verbal false limbs" that "save the trouble of picking out appropriate verbs." They add syllables with zero information. | Run a search-and-replace audit. "In order to" becomes "to". "Due to the fact that" becomes "because". "At this point in time" becomes "now". |
| V3 | Default passive voice | "The configuration is applied by running the command" / "The file should be edited before deployment" | Passive conceals the actor, which in instructions is usually the reader: the most important piece of information in the sentence. | Identify the actor and make them the subject. "Run the command to apply the configuration." Use passive only when the actor is genuinely irrelevant. |
| V4 | First-person plural overuse | "We will now see that...", "Let us consider the following example...", "As we discussed..." | The false "we" implies the writer and reader are co-discoverers, which is condescending: the writer already knows everything. It also distances the writer from responsibility for the instruction. | Replace with second person ("you") or imperative ("Run the command"). Reserve "we" for genuine organizational statements ("We recommend this approach"). |
| V5 | Inflated diction | "Utilize the API to facilitate the configuration process and leverage the existing infrastructure" | Orwell: words like `utilize`, `eliminate`, `liquidate` "are used to dress up simple statements." Latin roots lend an air of bureaucratic authority that alienates developers. | Use everyday English. "Use the API to configure the service." See voice-and-tone.md for the full replacement table. |
| V6 | Imprecise quantifiers | "Many users find that the performance is often improved by increasing the cache size" | Neither "many" nor "often" is actionable. The reader cannot calibrate the advice. | If you have data, cite it. If you do not, drop the claim or rephrase as a tested observation: "In our benchmarks, a 512 MB cache reduced latency by 40%." |
| V7 | Inappropriate familiarity | "Awesome! Now you've completed the installation." / "Of course, this is just one way to do it!" | Exclamations perform enthusiasm the reader does not share; "of course" implies the reader should already know, which is condescending. | Strike all exclamation points except in genuine warnings. Replace "Of course" and "Obviously" with nothing: if it is truly obvious, skip the sentence. |
| V8 | Inappropriate formality | "The configuration artifact pertaining to the aforementioned service must be modified prior to instantiation" | Over-formal register creates cognitive distance and signals the author is performing expertise rather than communicating it. | Read the sentence aloud. If you would not say it to a colleague, rewrite it. |

## AI-generated prose telltales

These 13 patterns are the strongest signals to a human reader that text is AI-generated. Each is a verbal tic that humans do not naturally use, but LLMs reach for them because their training corpora rewarded the pattern.

| # | Anti-pattern | Signature pattern | Why it sounds AI-generated | Human equivalent |
|---|---|---|---|---|
| A1 | The "delve" reflex | "Let's delve into the intricacies of..." / "We'll delve deeper into this topic" | A 2024 study of 14 million PubMed abstracts found "delve" spiked 9x after ChatGPT's launch: rare in human writing, statistical fingerprint of LLM output. | Humans say "look at", "explore", "go through", "examine". No human writer naturally reaches for "delve" in everyday technical prose. |
| A2 | The summary sandwich | Every section ends with "In summary, we covered..." / "To summarize, the key points are..." | LLMs are trained to produce completeness. Summaries at the end of every section are redundant: the reader just read the section. | Human writers summarize only at the end of multi-chapter works or when writing an executive summary. Section-by-section recap treats the reader as incapable of short-term memory. |
| A3 | The importance disclaimer | "It's important to note that..." / "It is worth noting that..." / "It should be noted that..." | A classic AI hedge that performs emphasis without providing it. The model hedges because it is uncertain; the phrase signals that without communicating why it matters. | A human writer who wants to emphasize something makes the sentence itself emphatic, or uses a callout box, or a bold phrase. They do not preface with meta-commentary about importance. |
| A4 | "Ensure that you" patronizing instruction | "Ensure that you have configured the settings before proceeding" / "Make sure to verify that the connection is active" | "Ensure" is bureaucratic redundancy. Every instruction already implies the reader should do it correctly. The phrase is a verbal tic from over-cautious technical writing training data. | Imperative: "Configure the settings before proceeding." The imperative already ensures. |
| A5 | Connector word overload | "Furthermore, this approach..." / "Moreover, it should be noted that..." / "Additionally, one must consider..." / "In essence, the framework..." | A 2024 arXiv corpus study found "furthermore", "moreover", and "additionally" appeared at 3 to 5x the frequency of human-written technical text. LLMs use transitional adverbs as structural glue. | Humans use "also", "and", "but", or simply a new sentence. Or they restructure to avoid needing a connector at all. |
| A6 | Em-dash explosion | "The configuration file, which can be found in the root directory, should be edited before, and only before, deployment" with em-dashes instead of commas | LLMs discovered the em-dash as a versatile pause marker and over-applied it. Human writers use em-dashes for strong interruptions or appositives; AI text uses them everywhere commas, colons, or parentheses would be more precise. | Use the right tool: comma for a mild clause, colon for elaboration, parentheses for incidental info. Per the user's rule, em-dashes are out entirely except in cited material. |
| A7 | The comprehensive-overview preamble | A 400-word introduction explaining what X is, its history, its use cases, and its limitations, before answering the question the heading promised. | LLMs optimize for completeness by design. The model wants to demonstrate knowledge before answering. The result is a preamble that delays the answer the reader came for. | State the answer first. Put the background in an "About X" explanation page that the reader can opt into. |
| A8 | Bullet-point paralysis | Every paragraph in the doc is exactly three bullets. Every bullet starts with a verb. Every bullet is 1.5 lines. No variation anywhere. | Excessive parallelism is a statistical signature of text generated by a system rewarded for consistency. Human documents have varied structure because structure follows content, not a template. | Use lists when items genuinely are list-like (discrete, enumerable, unordered). Use prose when ideas have causal or sequential relationships. Let content drive form. |
| A9 | Tone disclaimers | "This may be controversial, but..." / "Some readers may disagree, but..." / "While this is a matter of debate..." | LLMs hedge against pushback: they were trained on RLHF that penalized confident claims rated as wrong. In technical docs, tone disclaimers are almost always noise. | State the claim with appropriate evidence. If there is genuine controversy, describe the positions factually: "Approach A is preferred by X because...; Approach B by Y because..." |
| A10 | The non-answer apology | "I cannot provide specific advice on X, but here's what I can say..." | A direct RLHF artifact. Models were trained to refuse requests they cannot fulfill while still generating output. | Document what is true. If something is out of scope, say so in one sentence and link elsewhere. Do not spend 200 words apologizing for the 50-word answer. |
| A11 | Over-explanation of prerequisites | "Before we begin, ensure that you have a computer with internet access, a modern browser installed, and a basic understanding of command-line interfaces" | LLMs list every conceivable prerequisite including obvious ones because they cannot calibrate what the reader already knows. | List only non-obvious prerequisites, and be specific: "This guide assumes familiarity with `git` but not with `git rebase`." |
| A12 | Structural mirror symmetry | Every H2 section has exactly: one-paragraph intro, three bullets, one code block, one closing sentence. Repeated seven times. | Generated text defaults to template structures because templates were common in training data and were rewarded by RLHF evaluators who found them readable. The result reads like a form letter. | Let content decide structure. A section with one important fact gets one sentence. A complex comparison gets a table. |
| A13 | In-essence restatement | "In essence, what this means is that..." / "Essentially, the framework provides..." | An LLM's equivalent of clearing its throat before restating something it already said. Signals the model is about to paraphrase its own output. | If the first explanation was clear, stop. If it was not, rewrite the first explanation. Do not add a second version prefaced with "in essence." |

## How to use this catalog

Three ways:

1. __At review time:__ skim the headings. For any anti-pattern that pings recognition for your draft, jump to that row and apply the fix.
2. __At authoring time:__ keep the AI-ism section open in a second tab while drafting. Most AI-generated prose flagged here can be avoided by not writing it in the first place.
3. __As an evolving artifact:__ when you discover a new anti-pattern in your own work, add it. The catalog is meant to grow with the writer.

The script `scripts/Invoke-DocSelfReview.ps1` mechanically catches a subset of the anti-patterns here (most of category 3, parts of categories 1 and 2). The rest require reading.
